# Outpay v0.1 Beta

> **Release channel:** Beta
> **Semantic version:** `0.1.0-beta`
> **Release date:** Not verified
> **API stability:** Subject to change

No git tag or GitHub release exists for this version in the repository at
the time of writing. This page describes the state of the `main` branch at
commit `39a2c31` ("feat(checkout): implement customer management in
checkout process, adding customer creation and retrieval logic, and update
related database queries and tests"), which is the commit this entire
documentation set was generated from.

## Release summary

v0.1 Beta is Outpay's first documented release: a non-custodial checkout
and payment-verification platform for accepting native USDC on Base. It is
intended for **internal testing, staging, and controlled beta merchants**,
not as a drop-in production payments processor ŌĆö the repository's own
`README.md` states the payment-verification pipeline is "still undergoing
audit remediation."

- **Supported payment flow:** merchant creates a checkout (dashboard or
  public API) ŌåÆ customer pays via the hosted checkout page ŌåÆ Alchemy webhook
  and/or RPC reconciliation detects the transfer ŌåÆ the payment-matching
  worker confirms it ŌåÆ the merchant is notified via a signed webhook (or can
  poll the API).
- **Supported network:** Base mainnet only.
- **Supported asset:** native USDC only.
- **Usable features:** merchant onboarding, checkout creation (dashboard and
  API), hosted checkout, payment detection (webhook + RPC reconciliation),
  a single merchant-webhook event type (`checkout.paid`), API keys, an admin
  console, and real usage-metering accounting.
- **Experimental features:** end-to-end payment detection under real
  transaction volume has not been independently validated for production
  reliability.
- **Real-money readiness:** not recommended without further validation ŌĆö see
  [Known limitations](#known-limitations) below and the main documentation's
  [Known limitations](../OUTPAY_DOCUMENTATION.md#known-limitations) section.
- **SDK status:** Planned. No SDK exists; integrate via `curl`/`fetch`/
  `requests` against the documented REST API.

## Dashboard

Merchant dashboard, checkout list/detail, payments list, settings (profile,
payout wallet, logo, webhook, store status), and developer console (API
keys, webhook configuration and delivery history) are all fully
database-backed with no hard-coded fixture data standing in for real
functionality. The admin console (`/admin/**`) is a separate, internal-only
surface gated by a dedicated `admin_users` grant, independent of merchant
roles. A small number of marketing-page CTAs and the global error boundary's
retry button are non-functional ŌĆö see the main documentation's
[Dashboard features](../OUTPAY_DOCUMENTATION.md#dashboard-features) section
for the exact list.

## Hosted checkout

The public `/checkout/[id]` page displays amount, network, recipient
wallet, a QR code, a wallet deep-link, a live countdown, and a polled
payment status (`waiting`/`detected`/`paid`/`expired`). Polling only ŌĆö no
WebSocket/SSE. Expiry is enforced server-side on every read, not only by a
background sweep.

## Payment processing

Payment matching runs entirely server-side against integer USDC base
units (never floating point), with explicit handling for exact payment,
slight overpayment (tolerated within a configurable threshold), large
overpayment, underpayment, wrong token, wrong chain, and late arrival after
expiry. See the main documentation's
[Payment lifecycle](../OUTPAY_DOCUMENTATION.md#payment-lifecycle) section for
the full state machine, which is narrower than the state machine originally
proposed in `ARCHITECTURE.md`.

## Blockchain integration

Base mainnet, native USDC only. Three coexisting detection layers: a
real-time Alchemy Address Activity webhook, a recent-window RPC
reconciliation scan (~every 90 seconds), and a deep reconciliation scan
(~every 30 minutes), plus a dedicated confirmation-recheck cycle. Provider
failover between Alchemy (primary) and Chainstack (secondary) is
implemented with health-aware ordering.

## API and authentication

- Public versioned API: `/api/v1/checkouts` (create, read),
  `/api/v1/payments` (list). Bearer-token API-key authentication.
- Dashboard API: Better Auth session cookies; unversioned; not intended as a
  stable external contract.
- Admin API: a separate, database-gated authorization primitive.
- **Two different JSON error envelopes exist** across these surfaces ŌĆö see
  the main documentation's [Error handling](../OUTPAY_DOCUMENTATION.md#error-handling).

## Webhooks

Merchant webhooks are HMAC-SHA256 signed (`X-Outpay-Signature: v1=<hex>`),
retried up to 7 times over roughly 15 hours, and auto-disable an endpoint
after 3 consecutive full failure cycles. **Only the `checkout.paid` event
type is implemented** today, despite eight being documented in the original
architecture plan.

## Database and migrations

15 migrations (`0000` through `0014`) establish the full schema, including
a notable sequence where migration `0010` removed billing/contact tables
that migrations `0012` and `0013` later restored with stricter constraints.
`DATABASE_SCHEMA.md` is not fully in sync with the migration history as of
this release ŌĆö see the main documentation's
[Database model](../OUTPAY_DOCUMENTATION.md#database-model) for the
verified, migration-sourced schema.

## Deployment and infrastructure

Single Docker image (Bun 1.3.14), four Railway services documented
(`outpay-web` + three workers); a fifth documented worker
(`worker:alerts`) has no corresponding Railway config file in this
repository. Database backup/recovery is explicitly unverified per
`DEPLOYMENT.md` itself.

## Security

SHA-256-hashed, timing-safe-compared API keys; AES-256-GCM-encrypted
webhook signing secrets; dual-layer log redaction; thorough SSRF protection
on merchant-configured webhook URLs (including cloud-metadata-range
blocking); parameterized SQL throughout. No CORS policy is configured
anywhere in the application, and admin routes have no rate limiting. See the
main documentation's [Security](../OUTPAY_DOCUMENTATION.md#security)
section for the full audit.

## Developer experience

No official SDK; `curl`/`fetch`/`requests` examples throughout the
documentation. No documented local workflow for testing the real Alchemy
webhook path (a public tunnel must be set up manually). CI runs lint, type
check, migrate, validate, unit tests, and build on every push/PR; there is
no e2e step in CI.

## Known limitations

See the main documentation's full
[Known limitations](../OUTPAY_DOCUMENTATION.md#known-limitations) list. The
highest-impact items for this release:

1. Payment-verification production readiness is explicitly unresolved
   (README's own words).
2. Database backup/recovery is unverified.
3. Only one merchant-webhook event type is implemented.
4. Reorg handling is not implemented.
5. No merchant-facing remediation workflow for underpaid/overpaid/mismatched
   payments.
6. No CORS policy and no admin-route rate limiting.

## Breaking-change risk

The following areas should be considered **unstable** and may change before
v1.0:

- API routes and their versioning scheme (currently only `/api/v1/`
  exists; the dashboard/admin/public surfaces are entirely unversioned).
- JSON field names on every object documented in
  [JSON object reference](../OUTPAY_DOCUMENTATION.md#json-object-reference).
- Validation rules and error codes.
- The two divergent JSON error-response envelopes may be unified.
- Authentication headers and API-key format.
- Webhook event names and payload shape ŌĆö additional event types are
  expected, and the existing `checkout.paid` payload shape may be revised
  toward the nested envelope originally proposed in `ARCHITECTURE.md`.
- Database schema, including enum value sets (`checkout_status_enum`,
  `payment_status_enum`, etc.) and table structure.
- Payment/checkout status vocabulary ŌĆö three different status vocabularies
  already coexist across the internal enum, the public v1 API, and the
  hosted-checkout API; expect consolidation.
- Environment-variable names, particularly the currently undocumented ones
  (`OUTPAY_WORKER_CONCURRENCY`, `RECONCILER_*`, etc.) which may be
  formalized or renamed.
- Checkout and network identifiers, if/when multi-chain support is added.
- Error codes.
- Proposed future SDK methods (not yet implemented at all).

> Beta releases do not guarantee backward compatibility. Review the
> changelog before updating a deployed Outpay instance or merchant
> integration.
